# C4 Model — Azit Hub

> Diagramas de arquitetura em três níveis (contexto, containers, componentes), conforme item 11 das "Definições Prévias". Renderizam no GitHub (mermaid). Atualizado em 14/07/2026.

## Nível 1 — Contexto do sistema

```mermaid
graph TB
    OP["Operador Azit<br/>(comercial, análise, cobrança, diretoria)"]
    CLI["Cliente<br/>(motorista/comprador — indireto)"]
    HUB["<b>Azit Hub</b><br/>Plataforma de venda parcelada de veículos<br/>com reserva de domínio"]
    ASAAS["Asaas<br/>(cobrança: boleto/Pix, webhook)"]
    FUT1["Assinatura digital<br/>(futuro)"]
    FUT2["Birôs de crédito<br/>(futuro)"]
    FUT3["Rastreador/bloqueador<br/>(futuro)"]

    OP -->|"opera via navegador (desktop/mobile)"| HUB
    HUB -->|"emite cobranças"| ASAAS
    ASAAS -->|"webhook de pagamento (202 + fila)"| HUB
    CLI -->|"paga fatura"| ASAAS
    HUB -.-> FUT1
    HUB -.-> FUT2
    HUB -.-> FUT3
```

Princípio de fronteira: **Asaas executa, Azit controla** — toda lógica de negócio vive no Hub; integrações são executoras, nunca donas de regra. CRM e ERP completos serão integrações complementares futuras, sem aprisionar dados centrais.

## Nível 2 — Containers

```mermaid
graph TB
    subgraph VPS["VPS Hostinger — Docker Swarm + Traefik (TLS)"]
        FE["<b>Frontend</b><br/>React + Vite (SPA autenticada)<br/>app.azitmove.com.br"]
        BE["<b>Backend API</b><br/>NestJS + Fastify + Prisma<br/>api.azitmove.com.br"]
        PG[("PostgreSQL 16<br/>fonte da verdade")]
        RD[("Redis 7<br/>filas BullMQ")]
    end
    ASAAS["Asaas (externo)"]

    FE -->|"REST /api/v1 (JWT)"| BE
    BE --> PG
    BE -->|"jobs: webhook, régua, D-5, ativação"| RD
    BE -->|"API de cobranças"| ASAAS
    ASAAS -->|"webhook"| BE
```

Monorepo pnpm: `apps/backend`, `apps/frontend`, `packages/@azit/types` (enums compartilhados), `packages/@azit/utils` (cálculo financeiro puro, testado). Monolito modular — decisão validada em 13/07 (ADR-001).

## Nível 3 — Componentes do backend (módulos NestJS)

```mermaid
graph LR
    subgraph Plataforma
        AUTH[auth]
        ALC[alcada]

    subgraph Pessoas
        TIT[titular]
        CONTA[conta]
    end
    subgraph Originação
        FUNIL["funil<br/>lead → simulação → proposta<br/>→ formalização → ativação"]
        ORIG[originacao]
    end
    subgraph Ativos
        ATV[ativo]
        CC[centro-custo]
        OC[origem-capital]
    end
    subgraph Carteira
        CTR[contrato]
        COB["cobranca<br/>fatura + webhook"]
        REG[regua]
        OPS["operacoes<br/>renegociação, novação,<br/>quitação, sinistro, reajuste"]
        CRED[credito]
    end
    subgraph Capital
        INV[investimento]
    end
    PROD[produto]

    FUNIL --> TIT & ATV & CTR & SIM
    CRED --> APR & CTR
    OPS --> APR & COB
    CTR --> COB
    REG --> COB
    APR --> ALC
    CC --> ATV
    INV --> OC
```

Convenções entre componentes: services conversam via injeção Nest; efetivadores registrados no motor de aprovação por registry (`onModuleInit`) para evitar dependência circular; enums mapeados na borda do service; leituras respeitam soft delete (`findFirst`).
